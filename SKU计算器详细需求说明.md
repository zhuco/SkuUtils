# SKU计算器详细需求说明

## 1. 文档目的

本文档用于明确 SKU 计算器第一版的详细业务需求、计算规则、界面结构与数据设计，作为后续使用 Rust 开发 Windows 单机桌面软件的直接依据。

本版需求基于现有 [公式.xlsx](D:\python\SkuUtils\公式.xlsx) 的计算方式整理，并结合你补充的实际业务规则进行了扩展，目标是将当前 Excel 计算方式迁移为可保存历史商品数据、可配置参数、可批量计算多销售规格的本地桌面软件。

## 2. 项目范围

### 2.1 项目目标

开发一个仅在 Windows 桌面运行的 SKU 成本与建议售价计算器，具备以下能力：

- 保存商品基础资料
- 支持新增商品时选择多个计划销售规格
- 按销售规格自动计算成本和建议售价
- 支持根据不同重量区间自动匹配运费
- 支持统一设置平台扣点、其他费用、包装费用、运营费用
- 保存每个商品及每次计算结果的历史数据

### 2.2 使用对象

- 电商运营
- 采购人员
- 负责人或利润测算人员

### 2.3 运行环境

- 操作系统：Windows 10 / Windows 11
- 软件类型：本地单机桌面软件
- 开发语言：Rust
- 本地数据库：SQLite

## 3. 现有 Excel 口径梳理

当前 `公式.xlsx` 的核心计算口径如下：

### 3.1 基础利润公式

```text
利润 = 客单价 - 产品成本 - 平台扣点 - 其他 - 运费
毛利率 = 利润 / 客单价
```

### 3.2 ROI 相关公式

```text
ROI = 客单价 × 点击转化率 / PPC
保本ROI = 1 / 毛利率
保本PPC = 客单价 × 点击转化率 / 保本ROI
```

### 3.3 示例口径

Excel 中的示例值大致为：

- 客单价：15
- 产品成本：8.7
- 平台扣点：1.2
- 其他：0
- 运费：1.7
- 利润：3.4
- 毛利率：0.226666...
- 保本ROI：4.411764...

说明：

- 当前 Excel 以单一销售单位为主进行测算
- 新软件需要扩展为“一个商品可对应多个计划销售规格”
- 计算结果不仅要看利润，还要生成建议售价

## 4. 核心业务需求

## 4.1 商品资料管理

系统应支持维护商品基础资料，并保存历史商品数据。

### 4.1.1 新增商品字段

新增商品时，至少录入以下字段：

- 商品名称
- 产品成本
- 单件重量
- 计划销售规格

可扩展字段：

- 商品编码
- 分类
- 备注
- 创建时间
- 更新时间

### 4.1.2 历史数据要求

系统应支持：

- 保存所有已添加商品
- 查看商品历史资料
- 修改商品后保留最新数据
- 查看该商品的历史计算记录
- 基于历史商品资料再次一键计算

## 4.2 计划销售规格

### 4.2.1 默认内置规格

系统内置默认销售规格：

```text
X1、X3、X5、X10、X20、X30、X50、X100
```

### 4.2.2 选择方式

新增商品时，计划销售规格为多选。

示例：

- 商品 A 可选择 `X1、X3、X5`
- 商品 B 可选择 `X10、X20、X50`

### 4.2.3 可修改要求

销售规格规则必须支持后台设置和修改，包括：

- 新增规格
- 删除规格
- 修改规格名称
- 修改规格对应数量
- 启用或停用某个规格

建议数据结构：

- 显示名称，例如 `X3`
- 数量值，例如 `3`
- 排序号
- 是否启用

## 4.3 计算结果要求

商品添加完成后，系统应支持一键计算该商品所选全部计划销售规格的结果。

每个销售规格至少生成以下结果：

- 规格名称
- 对应销售数量
- 规格总成本
- 规格总重量
- 对应运费
- 平台扣点金额
- 其他费用
- 包装费用
- 运营费用
- 原始建议售价
- 最终建议售价
- 预计利润
- 预计毛利率

## 5. 计算规则设计

## 5.1 多规格计算基础逻辑

假设商品单件成本为 `unit_cost`，单件重量为 `unit_weight`，某销售规格数量为 `spec_qty`，则：

### 5.1.1 规格总成本

```text
规格商品成本 = unit_cost × spec_qty
```

### 5.1.2 规格总重量

```text
规格总重量 = unit_weight × spec_qty
```

### 5.1.3 运费匹配

```text
根据规格总重量落入的重量区间，自动匹配对应运费
```

例如：

- 0kg < 重量 <= 0.5kg，运费 2 元
- 0.5kg < 重量 <= 1kg，运费 3.5 元
- 1kg < 重量 <= 2kg，运费 5 元

最终以设置界面录入的数据为准。

## 5.2 成本构成

第一版建议将每个规格的总成本定义为：

```text
总成本 = 规格商品成本 + 运费 + 包装费用 + 运营费用 + 其他费用
```

说明：

- `规格商品成本` 来自单件成本乘以规格数量
- `运费` 根据规格总重量匹配重量区间
- `包装费用` 为系统设置项
- `运营费用` 为系统设置项
- `其他费用` 为系统设置项

## 5.3 平台扣点

平台扣点按售价比例计算：

```text
平台扣点金额 = 售价 × 平台扣点比例
```

例如：

- 平台扣点比例为 8%
- 售价为 19.9 元
- 平台扣点金额 = 19.9 × 0.08

## 5.4 利润与毛利率

### 5.4.1 预计利润

```text
预计利润 = 售价 - 总成本 - 平台扣点金额
```

### 5.4.2 毛利率

```text
毛利率 = 预计利润 / 售价
```

## 5.5 建议售价规则

你要求的建议售价规则为：

```text
建议售价 = 向上取整后 - 0.1 元
```

建议在系统内拆成两步：

### 5.5.1 原始建议售价

系统先按目标利润率反推一个“原始建议售价”。

推荐公式：

```text
原始建议售价 = 总成本 / (1 - 平台扣点比例 - 目标利润率)
```

说明：

- 该公式适用于平台扣点和目标利润率均按售价比例计算的情况
- 如果后续运营费用改成按售价比例，也需要纳入分母

### 5.5.2 最终建议售价

在原始建议售价基础上执行定价规则：

```text
最终建议售价 = ceil(原始建议售价) - 0.1
```

示例：

- 原始建议售价为 12.36，最终建议售价为 12.9
- 原始建议售价为 15.01，最终建议售价为 15.9
- 原始建议售价为 20.00，最终建议售价为 19.9

### 5.5.3 特别说明

上述规则会导致“整数价”被处理成“整数减 0.1”。

例如：

- 原始建议售价为 20.00
- 向上取整后为 20
- 最终建议售价为 19.9

这符合你当前提出的规则，但从业务上看可能出现最终售价低于原始建议售价的问题。为避免后续争议，建议在开发时提供两种可选规则：

#### 规则A：严格按当前要求

```text
最终建议售价 = ceil(原始建议售价) - 0.1
```

#### 规则B：更常见的电商尾数定价

```text
最终建议售价 = ceil(原始建议售价 - 0.1) + 0.9
```

第一版可先实现规则A，并在设置中保留后续扩展空间。

## 5.6 ROI 扩展指标

如果需要延续 Excel 的投流测算逻辑，可在结果页额外展示：

- 保本ROI
- 指定转化率下的保本PPC

公式沿用当前 Excel：

```text
保本ROI = 1 / 毛利率
保本PPC = 售价 × 转化率 / 保本ROI
```

第一版可做成可选展示项，不作为主流程必填项。

## 6. 设置中心需求

设置中心是本项目的关键模块，必须支持集中维护计算参数。

## 6.1 重量区间运费设置

系统应支持维护重量区间和对应运费。

### 6.1.1 字段建议

- 起始重量
- 截止重量
- 运费
- 备注
- 排序号
- 是否启用

### 6.1.2 规则要求

- 一个重量只能命中一个有效区间
- 区间不可重叠
- 支持新增、编辑、删除
- 支持启用或停用

### 6.1.3 计算要求

商品计算时，使用：

```text
规格总重量 = 单件重量 × 规格数量
```

再按总重量自动匹配运费。

## 6.2 平台扣点设置

系统应支持维护平台扣点比例。

### 6.2.1 字段建议

- 平台名称
- 扣点比例
- 是否默认
- 是否启用

### 6.2.2 使用方式

- 新增商品或计算时可选择平台
- 若未单独指定，默认取系统默认平台参数

## 6.3 其他费用设置

系统应支持设置“其他费用”默认值。

建议字段：

- 费用名称
- 金额
- 是否启用
- 备注

第一版可以先实现为全局固定金额。

## 6.4 包装费用设置

系统应支持设置包装费用默认值。

建议字段：

- 包装费用金额
- 是否启用
- 备注

第一版可以先实现为全局固定金额。

## 6.5 运营费用设置

系统应支持设置运营费用默认值。

建议字段：

- 运营费用金额
- 是否启用
- 备注

第一版可以先实现为全局固定金额。

## 6.6 目标利润率设置

因为建议售价需要反推，系统应提供默认目标利润率设置。

建议字段：

- 默认目标利润率
- 是否允许计算时临时修改

## 6.7 售价尾数规则设置

系统应预留售价尾数规则配置。

```text
向上取整后减 0.1 元
```

## 10. 技术实现建议

如果目标是纯 Rust Windows 桌面工具，第一版建议采用：

```text
Rust + egui/eframe + SQLite + rusqlite 或 sqlx + rust_decimal
```

推荐原因：

- 符合“Rust UI + Windows 桌面”的要求
- 表单、列表、设置页和结果页都能覆盖
- 单机本地数据库实现简单
- 适合快速做内部业务工具

## 11. MVP 范围

第一版建议只做以下范围：

- 商品新增、编辑、删除
- 销售规格多选
- 一键计算多个销售规格
- 重量区间自动匹配运费
- 平台扣点设置
- 包装费用、运营费用、其他费用设置
- 建议售价计算
- 计算历史保存与查询

暂不建议第一版实现：

- 多用户权限
- 云端同步
- 自动抓取平台数据
- 复杂图表
- 打印功能

## 13. 一句话结论

这个项目的核心不是“单个 SKU 的静态计算”，而是“商品主数据 + 多销售规格 + 可配置运费和费用规则 + 历史结果留存”的本地测算系统。第一版最合理的落地方式，是用 Rust 做 Windows 桌面程序，用 SQLite 保存商品、设置和历史结果，并严格沿用当前 Excel 的利润逻辑，再补上建议售价和批量规格计算能力。
