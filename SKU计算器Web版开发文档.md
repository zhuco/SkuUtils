# SKU计算器 Web 版开发文档

## 1. 文档目的

本文档基于当前项目已实现的桌面版功能与界面，整理出一份可直接指导 Web 版开发的完整方案。目标产品为：

- 一个可在线访问、免费使用的 SKU 计算工具
- 不依赖用户注册登录即可使用
- 用户数据默认只保存在浏览器本地
- 支持继续沿用当前桌面版的计算逻辑、界面结构和主要交互习惯

本文档重点不是重新发明一个新产品，而是把当前已经跑通的功能，平滑迁移为一个可部署到公网的 Web 工具。

## 2. 当前项目现状

### 2.1 当前技术形态

当前项目是一个 Rust 本地桌面程序，技术栈如下：

- 语言：Rust
- 桌面 UI：`eframe + egui`
- 本地存储：`SQLite`
- 时间处理：`chrono`

程序启动后会在当前目录创建或读取 `sku_calculator.db`，所有数据均保存到本地数据库文件。

### 2.2 当前已实现功能

根据当前源码，桌面版已经实现以下核心能力：

#### 2.2.1 商品基础录入

主界面支持录入：

- 商品名称
- 单件成本
- 单件重量
- 多个销售规格

其中商品名称不能为空，重量必须大于 0，成本必须为大于等于 0 的数字。

#### 2.2.2 销售规格管理

系统首次启动时会自动生成默认规格：

```text
X1、X3、X5、X10、X20、X30、X50、X100
```

当前支持：

- 勾选多个销售规格参与计算
- 新增自定义规格数量
- 删除已有规格
- 在历史回填时自动补齐缺失规格

#### 2.2.3 费用设置

当前设置页已支持以下参数：

- 平台扣点：仅支持百分比
- 包装成本：支持固定值或百分比
- 其它成本：支持固定值或百分比
- 运营费用：固定值
- 目标毛利率：百分比
- 重量区间运费规则：支持新增、删除、保存

#### 2.2.4 运费规则管理

运费规则按重量区间命中，单位为克 `g`。

当前规则要求：

- 至少保留一条规则
- 最小重量不能小于 0
- 最大重量必须大于最小重量
- 运费不能为负数
- 区间不能重叠
- 命中规则为：`重量 >= 最小值 且 重量 <= 最大值`

首次启动时会自动生成默认运费规则：

```text
0 - 450g      => 1.5
451 - 900g    => 1.8
901 - 1800g   => 2.1
1801 - 2600g  => 2.6
2601 - 4500g  => 5.0
```

#### 2.2.5 计算功能

当前支持两种操作：

- 仅计算：不保存商品，仅生成当前计算结果
- 添加并保存：保存商品、保存历史快照，并生成结果

计算结果按所选销售规格批量生成，当前输出字段包括：

- 商品名称
- 规格名称
- 商品成本
- 总重量
- 运费
- 包装成本
- 其它成本
- 运营费用
- 总成本
- 建议售价
- 毛利率
- 保本 ROI

#### 2.2.6 最近商品

当前主界面底部支持展示最近商品列表，能力包括：

- 显示最近 20 条商品
- 展示商品名称、成本、重量、规格、更新时间
- 双击可回填到主界面重新计算
- 支持删除商品

商品删除时会级联删除：

- 商品规格关联
- 该商品下的所有历史记录
- 对应的历史明细

#### 2.2.7 历史记录

当前历史窗口支持：

- 按商品名称模糊查询历史
- 默认读取最近 100 条历史记录
- 点击查看历史详情
- 双击记录回填到主界面重新计算
- 删除单条历史记录

历史记录保存的是快照，不只是商品主数据。当前已经保存了：

- 商品名称
- 单件成本
- 单件重量
- 所选规格数量列表
- 平台扣点
- 包装成本模式和值
- 其它成本模式和值
- 运营费用
- 目标毛利率
- 每个规格对应的计算结果明细

这意味着即使后续设置发生变化，历史明细仍可保留当时的计算结果。

### 3.4 当前 UI 总结

因此 Web 版不建议推翻重做，而建议保留现有产品结构，只做组件化和响应式改造。

## 4. Web 版产品定位

### 4.1 目标

将当前桌面版迁移为一个在线可访问的免费工具，满足以下要求：

- 用户打开网址即可使用
- 所有计算在浏览器本地完成
- 历史记录和商品数据保存在用户自己的浏览器中
- 服务端不保存用户业务数据
- 不需要账号体系
- 部署成本尽可能低

### 4.2 推荐产品形态

推荐做成一个前端单页应用 `SPA`，可选支持 `PWA`。

推荐原因：

- 当前功能全部属于本地计算和本地持久化，完全不需要后端参与
- 免费在线发布时，纯前端静态部署成本最低
- 用户数据不上传服务器，更符合“本地保存历史记录”的要求
- 可以天然规避账号、权限、数据库运维、服务端费用等问题

### 4.3 Web 版 MVP 核心原则

- 第一版不做登录
- 第一版不做云端同步
- 第一版不做服务端数据库
- 第一版只做静态托管 + 浏览器本地存储
- 第一版优先完整迁移当前桌面版已实现功能

## 5. Web 版总体方案

### 5.1 推荐技术栈

推荐采用以下前端技术栈：

- 框架：React
- 语言：TypeScript
- 构建工具：Vite
- UI 组件：Ant Design
- 本地数据库：IndexedDB
- IndexedDB 封装：Dexie
- 状态管理：Zustand
- 表单校验：Zod
- 单元测试：Vitest
- 端到端测试：Playwright

### 5.2 为什么不推荐首版上后端

如果产品要求是“免费在线使用 + 用户自己本地保存历史”，那么后端并不能带来核心价值，反而会增加：

- 服务器费用
- 数据库维护成本
- 用户隐私责任
- 登录、鉴权、风控复杂度
- 网络失败带来的可用性问题

当前 SKU 计算逻辑本身是纯函数型业务，非常适合放在前端运行。

### 5.3 部署形态

推荐部署方式：

- `Vercel`
- `Cloudflare Pages`
- `Netlify`
- `GitHub Pages`

如果需要国内访问体验更稳定，可后续再切换到国内静态托管平台或自有 CDN。

## 7. 浏览器本地存储方案

### 7.1 存储原则

用户明确希望“自己本地保存历史记录”，因此 Web 版必须采用浏览器本地持久化。

推荐原则：

- 业务数据统一保存在 `IndexedDB`
- 少量 UI 偏好可保存在 `localStorage`
- 不向服务端上传商品和历史数据

### 7.2 为什么选择 IndexedDB

不建议只用 `localStorage` 保存历史数据，原因如下：

- `localStorage` 容量小
- 查询能力弱
- 不适合保存大量历史记录和结果明细
- 不适合保存结构化列表数据

`IndexedDB` 更适合当前场景：

- 可保存大量结构化数据
- 适合商品、规格、历史明细等对象数据
- 能按索引搜索
- 可离线工作

### 7.3 推荐数据模型

Web 版不必完全照搬 SQLite 关系表结构，但要保留当前业务语义。推荐采用“前端友好型”结构。

## 8. 计算规则设计

Web 版必须严格沿用当前桌面版算法，避免出现同一输入在不同端结果不一致。

### 8.1 变量定义

- `unitCost`：单件成本
- `unitWeight`：单件重量
- `specQty`：规格数量
- `platformRate`：平台扣点比例
- `packageRate`：包装费百分比
- `otherRate`：其它费百分比
- `targetProfitRate`：目标毛利率

### 8.2 商品成本

```text
goodsCost = unitCost * specQty
```

### 8.3 总重量

```text
totalWeight = unitWeight * specQty
```

### 8.4 运费命中规则

```text
shippingFee = 第一条满足
weight >= weightMin 且 weight <= weightMax
的运费规则对应金额
```

若未命中任何运费规则，则当前桌面版返回 `0`。

### 8.5 固定成本基数

当包装和其它费用为百分比时，它们先不进入固定成本基数。

```text
fixedCostBase =
  goodsCost
  + shippingFee
  + operationFee
  + packageFeeFixed
  + otherFeeFixed
```

### 8.6 原始建议售价

```text
denominator =
  1
  - platformRate
  - packagePercent
  - otherPercent
  - targetProfitRate

rawSuggestedPrice =
  denominator > 0 ? fixedCostBase / denominator : 0
```

### 8.7 最终建议售价

当前桌面版的尾数规则是：

```text
finalSuggestedPrice =
  rawSuggestedPrice > 0
    ? max(ceil(rawSuggestedPrice) - 0.1, 0.1)
    : 0
```

例如：

- `12.36 => 12.9`
- `15.01 => 15.9`
- `20.00 => 19.9`

Web 版第一版应与当前逻辑完全一致，不要自行修改定价规则。

### 8.8 包装和其它费用

```text
packageFee =
  fixed 模式 => packageFeeValue
  percent 模式 => finalSuggestedPrice * packagePercent

otherFee =
  fixed 模式 => otherFeeValue
  percent 模式 => finalSuggestedPrice * otherPercent
```

### 8.9 总成本

```text
totalCost =
  goodsCost
  + shippingFee
  + packageFee
  + otherFee
  + operationFee
```

### 8.10 平台扣点

```text
platformFee = finalSuggestedPrice * platformRate
```

### 8.11 预计利润

```text
estimatedProfit =
  finalSuggestedPrice - totalCost - platformFee
```

### 8.12 毛利率

```text
profitRate =
  finalSuggestedPrice > 0
    ? estimatedProfit / finalSuggestedPrice
    : 0
```

### 8.13 保本 ROI

```text
breakEvenRoi =
  profitRate > 0
    ? 1 / profitRate
    : 0
```

## 10. 交互与状态流

### 10.1 首次打开页面

流程如下：

1. 初始化 IndexedDB
2. 检查默认规格是否存在
3. 检查默认运费规则是否存在
4. 检查默认设置是否存在
5. 读取最近商品
6. 读取最近历史

### 10.2 点击“仅计算”

流程如下：

1. 校验表单
2. 读取已选规格
3. 逐规格计算结果
4. 展示结果表
5. 不写入商品数据
6. 不写入历史数据

### 10.3 点击“添加并保存”

流程如下：

1. 校验表单
2. 根据商品名称查重或按 ID 更新
3. 保存商品
4. 根据当前设置生成快照
5. 保存历史记录
6. 刷新最近商品
7. 刷新历史记录
8. 展示结果表

### 10.4 从最近商品回填

流程如下：

1. 读取商品数据
2. 回填名称、成本、重量、规格
3. 自动重新计算

### 10.5 从历史记录回填

流程如下：

1. 读取历史记录
2. 回填商品名称、成本、重量、规格
3. 如规格在当前规格库中不存在，则自动补建
4. 自动重新计算

这一点应继续沿用当前桌面版行为。

## 11. 非功能需求

### 11.1 性能

- 首屏可交互时间小于 2 秒
- 单次多规格计算时间小于 100 毫秒
- 历史 1000 条以内检索应无明显卡顿

### 11.2 可用性

- 桌面端宽度 `>= 1280px` 时保持三段式工作台布局
- 平板端使用上下布局
- 手机端将“设置”和“历史”切换为独立页面或全屏抽屉

### 11.3 可靠性

- 任何本地持久化写入失败时，界面要有明确错误提示
- 导入数据前先做 schema 校验
- 删除操作必须二次确认

### 11.4 可维护性

- 计算逻辑必须与 UI 分层
- IndexedDB 操作必须集中封装
- 所有金额、百分比格式化逻辑统一走工具函数

## 16. 最终结论

当前项目已经具备很好的 Web 化基础。它的核心业务并不依赖服务端，主流程也已经被桌面版验证过，因此最合理的 Web 版路线不是“前后端重平台化”，而是：

```text
React + TypeScript + Ant Design + IndexedDB
静态部署到公网
所有计算在前端执行
所有商品和历史默认保存在用户浏览器本地
```

这样做有四个直接好处：

- 开发成本低
- 部署成本低
- 免费开放使用压力小
- 与“用户自己本地保存历史记录”的目标完全一致

如果按本文档推进，第一版 Web 工具可以在产品结构、计算结果和使用习惯上高度继承当前桌面版，同时又满足在线访问、免费使用和本地留存数据的目标。
