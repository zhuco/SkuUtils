# SKU计算器产品与技术方案

## 1. 项目目标

开发一个仅运行于 Windows 桌面的单机版电商 SKU 计算器，使用 Rust 语言实现，具备本地数据库存储能力，并提供桌面图形界面，用于快速计算 SKU 的成本、预计售价、毛利率及相关经营指标。

该软件面向电商运营、采购、财务测算等场景，目标是让用户能够：

- 快速录入 SKU 的采购、物流、包装、平台扣点等成本项
- 自动计算单件总成本、建议售价、预计毛利、毛利率
- 保存并查询 SKU 历史测算记录
- 支持不同平台、不同计价模板下的重复测算
- 在不依赖网络服务的前提下完成本地使用

## 2. 产品定位

### 2.1 使用场景

- 新品上架前的价格测算
- 老品调价前的利润复核
- 多平台之间的售价策略比较
- 不同采购价、运费、汇率条件下的模拟测算

### 2.2 目标用户

- 电商运营人员
- 采购人员
- 财务或成本核算人员
- 小型电商团队负责人

### 2.3 部署方式

- 仅支持 Windows 桌面运行
- 单机本地安装
- 本地数据库保存数据
- 不依赖浏览器
- 不强制依赖网络

## 3. 核心功能需求

### 3.1 SKU基础信息管理

支持维护 SKU 的基础主数据，建议字段如下：

- SKU编码
- 商品名称
- 商品分类
- 品牌
- 规格型号
- 颜色
- 尺寸
- 重量
- 体积
- 供应商
- 备注

### 3.2 成本项录入

每个 SKU 支持录入多个成本项目，建议包括：

- 采购单价
- 国内运费
- 国际运费
- 包装成本
- 人工处理费
- 仓储分摊成本
- 平台佣金比例
- 支付手续费比例
- 推广费比例
- 税费
- 汇率
- 其他成本

系统应支持：

- 固定金额成本项
- 按比例计算的成本项
- 选填和必填字段控制
- 通过模板快速带出默认值

### 3.3 自动测算功能

系统应至少支持以下计算结果：

- 单件总成本
- 预计售价
- 预计毛利
- 毛利率
- 净利润
- 净利率
- 平台到手金额
- 保本售价

建议支持两种常见测算方式：

1. 已知售价，反推利润
2. 已知目标利润率，反推建议售价

### 3.4 平台与模板管理

支持维护不同平台或渠道的测算模板，例如：

- 淘宝
- 天猫
- 京东
- 拼多多
- 抖音
- 亚马逊
- 独立站

每个平台模板可配置：

- 平台名称
- 佣金比例
- 支付手续费比例
- 默认推广费比例

### 3.5 测算记录管理

每次测算都应支持保存，便于后续追溯。

建议能力：

- 保存测算记录
- 按 SKU 查询历史记录
- 按平台查询记录
- 按日期区间筛选
- 复制已有记录并重新测算
- 导出测算结果为 Excel 或 CSV

### 3.6 报表与对比

建议提供基础分析能力：

- 同一 SKU 在不同平台下的利润对比
- 同一 SKU 在不同售价下的利润区间对比
- 成本项构成展示
- 最近测算记录列表

### 3.7 系统功能

- 本地数据库初始化
- 数据备份与恢复
- 基础参数配置
- 主题设置
- 日志记录

## 4. 核心计算规则建议

### 4.1 基础公式

以下为建议公式，后续可按业务细化：

#### 4.1.1 单件总成本

```text
单件总成本 = 采购单价
          + 国内运费
          + 国际运费
          + 包装成本
          + 人工处理费
          + 仓储分摊成本
          + 税费
          + 其他固定成本
```

#### 4.1.2 平台扣费

```text
平台扣费 = 售价 × 平台佣金比例
支付手续费 = 售价 × 支付手续费比例
推广费用 = 售价 × 推广费比例
```

#### 4.1.3 到手金额

```text
平台到手金额 = 售价 - 平台扣费 - 支付手续费 - 推广费用
```

#### 4.1.4 毛利

```text
毛利 = 平台到手金额 - 单件总成本
```

#### 4.1.5 毛利率

```text
毛利率 = 毛利 / 售价
```

#### 4.1.6 反推建议售价

当用户输入目标毛利率时，可用以下思路反推：

```text
建议售价 = 固定成本 / (1 - 目标毛利率 - 平台佣金比例 - 支付手续费比例 - 推广费比例)
```

说明：

- 上式仅适用于主要扣费均按售价比例计算的情况
- 若税费、运费或其他费用存在阶梯规则，需要单独扩展算法
- 实际开发中建议将计算逻辑拆分为规则引擎或独立服务层，便于维护

## 6. 技术方案建议

## 6.1 总体技术选型

结合需求，推荐技术栈如下：

- 开发语言：Rust
- 桌面 UI：Tauri + Rust 后端 + 前端界面
- 数据库：SQLite
- ORM / 数据访问：`sqlx` 或 `rusqlite`
- 配置管理：TOML 或 JSON
- 日志：`tracing` + `tracing-subscriber`
- 导出能力：CSV 优先，Excel 可后续扩展

### 6.2 UI框架建议

这里有两种主流方案：

#### 方案A：Tauri + 前端界面

优点：

- 桌面应用体验成熟
- 安装包相对轻量
- 生态完善
- 前端界面更容易做复杂表单和数据表格

缺点：

- UI 部分不属于纯 Rust
- 需要同时维护前端与 Rust 后端

适用性判断：

- 如果你更重视界面易用性、表格能力和后续扩展，优先选这个方案

#### 方案B：纯 Rust UI，例如 `egui` / `iced`

优点：

- UI 与业务逻辑都可使用 Rust
- 技术栈统一
- 单文件桌面工具体验较直接

缺点：

- 表单和复杂管理界面的成熟度通常不如 Web UI
- 某些组件能力和美观度需要自行打磨

适用性判断：

- 如果你明确要求“尽量纯 Rust UI”，推荐优先考虑 `egui`

### 6.3 本项目推荐方案

因为你提出“需要 Rust 的 UI，只在 Windows 桌面运行”，这里给出两级推荐：

#### 推荐方案一：`egui` + `eframe` + SQLite

适合：

- 希望 UI 和业务都使用 Rust
- 项目以内部工具为主
- 更看重开发简单和技术栈统一

优点：

- 全 Rust 实现
- 开发门槛低于自绘原生 UI
- 本地工具类软件足够实用
- 与 SQLite 集成简单

#### 推荐方案二：Tauri + Rust 后端 + Web 前端

适合：

- 后续要做更复杂的数据表格、筛选、图表、导出
- 需要更现代的界面交互

如果你的核心要求是“Rust 语言开发”和“Windows 本地桌面工具”，同时不强调前端必须也是 Rust，那么 Tauri 的综合体验更强。

如果你的核心要求是“UI 也要尽量使用 Rust”，那么第一版建议直接采用：

```text
Rust + egui/eframe + SQLite + sqlx
```

## 10. 关键开发难点

### 10.1 金额精度

金额计算不建议直接使用浮点数进行最终财务结果存储。

建议：

- 内部计算尽量使用定点数方案
- 或使用高精度十进制库
- 数据库存储时统一约定精度

Rust 可考虑：

- `rust_decimal`

### 10.2 规则扩展

电商场景中，实际平台费用规则可能存在：

- 阶梯费率
- 满减活动
- 包邮门槛
- 不同仓发货规则

建议在架构上将“计算规则”与“界面录入”分离，避免后续修改牵动整个系统。

### 10.3 数据备份

单机软件必须考虑用户误删或系统异常。

建议增加：

- 手动备份数据库
- 自动定时备份
- 启动时检测数据库完整性

## 12. MVP版本建议范围

为了尽快做出可用版本，建议 MVP 只做以下内容：

- SKU管理
- 成本录入
- 售价与毛利计算
- 模板管理
- 测算记录保存与查询
- CSV导出
- SQLite本地数据库

暂不建议第一版就做：

- 多用户权限
- 联网同步
- 云端接口
- 自动抓取平台数据
- 复杂 BI 报表

## 13. 推荐技术结论

如果你要尽快开工，并且希望“Rust + Windows 桌面 + 本地数据库 + UI 也尽量是 Rust”，建议采用以下技术方案：

```text
Rust
+ egui / eframe
+ SQLite
+ sqlx 或 rusqlite
+ rust_decimal
+ tracing
```

推荐理由：

- 纯本地桌面工具实现简单
- 技术栈统一
- 开发速度较快
- 适合内部业务工具
- 后续可以逐步扩展导出、模板、备份等功能
